TIBD数据库概述

TIKV是行存储，TIflash是列存储

TIDB Server

Protocol Layer协议层处理客户端连接

Parse、Compile做sql语句的解析与编译，生成的执行计划交给Executor

语法树传给Compile模块

关系型数据库表转换为key_value，为保证在数据库作为唯一索引，表号加主键

Region默认为96M，当达到144M的时候会有一个split变成两个，所以整个TIDB数据库是以Region为单位的分布式存储在TIKV里边。

每一层调用下一层的方法。

分成两种

第一种

​ 复杂（涉及多个表）的sql会走DistSql这条路。

​ 复杂的语句，如表连接、嵌套查询等等。抽象出DistSql负责封装TItv的请求提供一个简单的select方法，所以不 管是什么复杂的sql经过DistSql之后都会变成单表的计算任务。

​ 简而言之将对多个的操作转化对单个表一个个操作的组合。

​ 所以一个DistSql涉及一个表但是可能涉及多个Region。

第二种

​ 简单（单个表）的sql会走KV这条路。

最后都是经过TIKV client向TIKV集群发送读写请求。

如果有事务相关操作executor会启动Transaction模块，Transaction负责两阶段提交。通过pd client与PD交互获取时间戳，得到事务提交的时间。

同一时间只能有一个TIDB server进行DDL操作。此时那个TIDB server叫owner

start job放DDL语句workr取语句

TIDB定期清理一些历史数据，这个就叫做GC，会有TIDB server被选为GC leader，

操作历史数据会保留GC life time（默认10分钟）时间。

不能超过64M

租约内只能读不能写
